# 项目优化记录 - 01_应用层双向ACK与可靠投递

---

## 1. 改进点介绍
在一对一即时通讯场景下，保障消息投递的**“绝对可靠（不丢消息）”**和**“幂等去重与防乱序”**。
*   **改进前问题**：原有的消息发送是单向、无反馈的。如果接收端因为网络瞬间抖动下线或处于弱网“假死”状态，服务器下发的消息就会直接丢失，而发送端和服务器都无法感知这一状态，导致离线消息转存失败，聊天体验较差。
*   **优化目标**：实现从“发送端 $\to$ 服务端 $\to$ 接收端”的全链路消息投递确认与重传逻辑，保证消息在任何恶劣网络环境下“零丢失、零重复、不乱序”，并能自动识别并物理清理因网络异常导致的假死连接。

---

## 2. 使用的技术栈
*   **网络与并发编程**：C++17, POSIX TCP 套接字, Linux Epoll (边缘触发 ET 模式), 多线程 (`std::thread`, `std::mutex`, `std::lock_guard`, `std::atomic`).
*   **协议与序列化**：Protobuf (Protocol Buffers v3).
*   **客户端框架**：Qt 6.10 (QTcpSocket, QTimer, QMap, QList).
*   **分布式架构设计**：基于雪花算法 (Snowflake ID) 的全局分布式消息生成, 应用层双向 ACK 超时确认与重传。

---

## 3. 改进思路与逻辑
我们采用**“应用层双向 ACK 与滑动窗口”**架构打通了可靠消息投递闭环：

### 3.1 协议升级
*   扩展 Protobuf 协议，在单聊请求 `OneChatRequest` 中加入 64 位全局唯一消息 ID `msg_id`。
*   新增 `MsgSendAck` 和 `MsgRecvAck` 报文体，分别代表“服务端接管确认”和“接收端收到确认”。

### 3.2 客户端逻辑 (发送方 A 与 接收方 B)
1.  **全局雪花 ID 生成**：A 发送消息前，通过本地封装的 `Snowflake` 生成全局唯一递增的 `msg_id`。
2.  **发送与重传机制**：A 将消息存入本地的 `pendingSendAckMap_` 重传映射表（记录发送时间与重发次数），并开启重发定时器（每 2 秒扫描一次）。
    *   若 3 秒内未收到服务端的 `MSG_SEND_ACK`，执行重发。
    *   若重发 3 次仍未响应，判定发送失败，更新 UI 气泡显示 `[发送失败 ⚠️]`。
    *   收到服务器的 `MSG_SEND_ACK` 后，将该消息移出队列，更新 UI 气泡为 `[已送达]`。
3.  **接收去重与自动回复**：B 收到消息后，**立即**向服务端自动回复 `MSG_RECV_ACK`。为了防止 B 的回复在网络中丢失导致服务端重传，B 内部维护一个大小为 1000 的已收消息 ID 滑动窗口：
    *   若收到重复 `msg_id` 的消息，B 在内存直接丢弃，不重复在 UI 渲染，但仍向服务端补发 `MSG_RECV_ACK` 以消除重传。
    *   若为新消息，则正常渲染至 UI。

### 3.3 服务端逻辑 (`ChatServer` 转发保障)
1.  **接管答复**：服务端收到 A 的消息后，立即向 A 发送 `MSG_SEND_ACK`，告知服务器已接管，发送端即可清除未确认标志。
2.  **本地转发与重传管理**：
    *   如果 B 连接在本台服务器：发送消息给 B，同时在业务层加锁将该消息存入 `_pendingRecvAckMap` 重传队列中，开启守护线程进行每 2 秒一次的重传扫描。
    *   若超时 3 秒未响应，重新向 B 发送该包；重传 3 次依旧超时后，判定 B 网络异常处于“假死连接”，服务端将该消息转存至 MySQL `OfflineMessage` 表，并通过新封装的 `TcpConnection::shutdown()` 接口强制切断并注销死连接，执行 `clientCloseException` 清理路由。
    *   如果收到 B 发送的 `MSG_RECV_ACK`，则加锁从 `_pendingRecvAckMap` 中擦除对应消息，重传终止。
3.  **跨服路由转发**：如果 B 连接在其他服务器上，本台服务器将消息发布到 Redis 中。B 所在的服务器在接收到订阅回调后，同样完成上述“本地转发与重传管理”的流程。
4.  **下线消息清退**：在用户下线或异常断开时，遍历重传队列，将其名下的待接收确认消息全部转存至 MySQL 离线消息，防止重传包写入已关闭的 socket。

---

## 4. 改进完成后的作用
1.  **消息丢失率为零**：即使在极端断网、强行拔网网线、进程假死等弱网异常下，系统也能确保未送达的消息最终转化为离线消息进行持久化，保证“消息必达”。
2.  **消除了网络假死占用的服务器资源**：服务器主动关闭超时不确认的死连接，提高了网络吞吐与系统整体稳定性。
3.  **客户端交互体验升级**：气泡状态由“发送即送达”变为清晰的“发送中”、“已送达”以及“发送失败（红叹号）”的可视化流转。

## 5. 对应的面试内容

#### 🙋‍♂️ 提问：既然 TCP 已经是可靠传输协议（有三次握手、重传和滑动窗口），为什么 IM 系统还需要在应用层设计双向 ACK？
*   **回答**：
    “TCP 的可靠传输只能保证消息安全送达**操作系统的内核接收缓冲区**，并不等于消息已经被应用层正确读取和处理。
    如果接收方的程序发生死锁、内存写满或者在读取缓冲区前瞬间崩溃，TCP 连接虽然在内核态正常发送了 ACK，但应用层实际上丢失了该消息。
    此外，若由于基站切换或拔掉网线导致断网，TCP 连接可能需要十几分钟甚至更久才能触发 Keep-Alive 超时判定。在此期间，服务端若盲目相信 TCP，就会误以为消息已送达而丢弃它。因此，必须引入应用层 ACK，只有当接收方的业务逻辑层显式答复了确认包，服务端才认为发送成功，否则超时重传并转存离线，这是实现‘消息零丢失’的唯一方案。”

#### 🙋‍♂️ 提问：在引入应用层重传后，如何解决消息重复投递的问题？
*   **回答**：
    “重传机制必定带来消息重复的风险（例如接收端 B 正常收到了消息，但回复给服务端的 ACK 报文在网络中丢失，导致服务端重传）。
    我们通过**幂等性消息去重**解决此问题。我们为每条消息分配一个基于雪花算法生成的全局唯一 `msg_id`。接收端客户端在内存中维护一个大小为 1000 的 `msg_id` 滑动窗口。每当收到消息，去窗口中匹配 ID：
    如果已存在，代表是重复投递的消息，客户端直接丢弃该数据，不重复在 UI 上渲染气泡，从而保证幂等性；但同时，客户端依然会向服务端补发该消息的 `MSG_RECV_ACK`，帮助服务端及时关闭其重传定时器，保持两端状态同步。”

## 6. 简历内容书写模板

*   **项目描述/核心成果表述**：
    1.  **应用层双向 ACK 与消息可靠投递**：自主设计并实现了一套**应用层双向确认（Double ACK）与超时重传控制机制**，确保消息全链路“零丢失”；在客户端引入 **Snowflake (雪花算法)** 实现全局分布式消息唯一 ID，并在接收端设计了 **1000 大小的幂等去重滑动窗口**，有效解决了消息重复渲染与弱网抖动下的**时序一致性（防乱序）**。
    2.  **高并发状态机与假死连接物理剔除**：在服务端（基于 Epoll 非阻塞 Socket）设计了重传守护扫描线程，结合状态路由自动发现网络异常状态；针对 3 次重发超时未响应的“假死长连接”，设计了 **MySQL 离线消息自动退路转存**与 **`TcpConnection::shutdown()` 强制套接字注销断开逻辑**，彻底消除了假死套接字对系统资源的占用。
