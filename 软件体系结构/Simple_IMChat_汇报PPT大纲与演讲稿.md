### Page 2：选题背景与设计痛点
*   **【视觉设计】**：

        2.  *网络不稳定*：移动网络瞬断引起的消息丢失与乱序。

    *   **右栏（项目定位）**：`Simple_IMChat` 致力于在应用层解决消息高可靠投递，实现多节点无状态水平扩容，提供自愈式高可用保障。
*   **【演讲配音稿】**：
    > “在互联网高并发场景下，即时通讯系统面临着极大的质量挑战。首先是单机如何承载数万用户的长连接；其次是移动网络切换导致的网络瞬断，极易引起消息丢失或乱序；最后是单点性能瓶颈，系统必须具备横向扩容能力。针对这些痛点，我设计并实现了 Simple_IMChat 系统，作为本次大作业的研究对象。”

### Page 3：混合体系结构风格设计
*   **【视觉设计】**：
    *   展示**“异构混合体系结构风格拓扑图”**（即报告中的混合风格总架构图，突出表现层、网关层、业务层、仓库层）。
    *   在四种风格旁边标注关键词：
        1.  *C/S 风格*：胖客户端（Qt）分离展现逻辑，服务端专注于高并发计算。
        2.  *分层风格*：客户端与服务端各自的内部 3-4 层纵向依赖切分。
        3.  *事件驱动/发布订阅*：单机 Epoll Reactor + 跨服 Redis Pub/Sub。
        4.  *仓库风格*：共享在线状态哈希表与 Timeline 持久化仓库。
*   **【演讲配音稿】**：
    > “在体系结构风格上，本系统没有采用单一风格，而是设计了一个‘异构混合体系结构’。
    > 它整体呈分布式客户机/服务器（C/S）格局，客户端分担了大量的展示与可靠确认逻辑。
    > 系统内部严格遵循分层风格，保证单向依赖和模块解耦。
    > 同时，在并发处理上，系统单机内部采用基于 Linux Epoll 的 Reactor 事件驱动风格，而在跨节点通信上则采用 Redis 的发布-订阅风格。最后，系统的数据流向是以共享的 Redis 状态网关和 MySQL 历史 Timeline 仓库为核心，体现了仓库风格。”

### Page 5：“4+1” 视图之 过程视图（可靠消息投递时序）
*   **【视觉设计】**：
    *   展示**“应用层双向 ACK 与滑动窗口去重的时序流转图”**（Mermaid 时序图）。
    *   用红色粗体圈出：*发送端待确认队列* $\rightarrow$ *超时重传* $\rightarrow$ *接收端滑动去重窗口*。
*   **【演讲配音稿】**：
    > “过程视图关注系统的并发和动态消息时序，这是本设计最核心的安全与可靠保障。
    > 传统 TCP 连接虽是可靠的，但当网络瞬断时，客户端可能无法及时感知，从而引发丢包。
    > 为此，我们在应用层引入了双向确认机制。发送端发出消息后将其放入‘待确认队列’，并启动 3 秒重传定时器。服务端收到后先进行滑动窗口去重，确保不重复入库渲染，随后回复 ACK。如果发送方超时未收到 ACK，则触发自动重传，直至收到确认或达到重传上限。这为系统提供了At-Least-Once的应用层可靠投递保障。”

### Page 6：“4+1” 视图之 过程视图（分布式路由与多线程）
*   **【视觉设计】**：
    *   展示**“跨节点 Pub/Sub 路由转发时序图”**。
    *   用箭头醒目标注：*Server1 $\rightarrow$ 查询 Redis 路由表 $\rightarrow$ Redis PUBLISH $\rightarrow$ Server2 收到订阅并推送到 UserB*。
*   **【演讲配音稿】**：
    > “在多节点集群环境下，用户可能连接在不同的服务器实例上。
    > 我们的过程视图还定义了跨节点的路由寻址流程：当 Server1 收到发送给 UserB 的消息时，发现 UserB 并不在本地，它会快速检索 Redis 共享状态网关。一旦确定 UserB 登录在 Server2 上，Server1 会向 Redis 对应的实例通道发布 PUBLISH 消息。Server2 通过长订阅接收到该广播，并在本地套接字表中找到 UserB 完成最终推送，从而抹平了多物理节点的物理差异。”

    > 评估涵盖了可用性、性能和可伸缩性三大核心指标。我们识别了其中的敏感点与权衡点，并对核心架构决策进行合理性论证。例如，瞬断重传保障了高可用，但带来了一定的内存和带宽折中；异步批量落盘极大地提升了系统的并发性能，但牺牲了一定的即时落盘强一致性。这些设计权衡都是为了在大高并发场景下获得最佳的系统表现。”

### Page 9：架构优化亮点总结
*   **【视觉设计】**：
    *   三个图文卡片，分别展示系统的核心亮点：
        1.  *Lazy Reconnect 数据库自愈*：解决容器启动网络断连，避免人工重启。
        2.  *Timeline 增量离线拉取*：秒级恢复离线历史，避免海量同步风暴。
        3.  *Epoll Reactor 线程解耦*：I/O 主线程与业务线程池解耦，实现单机高吞吐。
*   **【演讲配音稿】**：
    > “总结一下 Simple_IMChat 架构上的三个核心亮点：
    > 第一，是自研连接池的懒重连机制，当数据库遇到抖动或容器颠倒拉起时，无需人工重启，毫秒级自愈。
    > 第二，是 Timeline 增量同步架构，用户离线后上线，服务器仅基于 Redis ZSet 进行极速区间截取，耗时极低。
    > 第三，是 Epoll Reactor 读写与数据库存盘的多线程完全解耦，磁盘慢 I/O 不会影响高并发网络响应。”
